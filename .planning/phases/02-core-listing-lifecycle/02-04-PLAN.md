<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser file picker → CreateListingForm | User-supplied file content crosses into client-side validation |
| CreateListingForm → Vercel Blob CDN | File bytes cross from browser to Vercel Blob (bypassing Next.js server) |
| CreateListingForm → POST /[token]/api/listings | JSON body crosses from browser to Route Handler |
| Browser localStorage | editToken written to localStorage — accessible to same-origin scripts |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04-A | Tampering | Uploading non-image files | mitigate | `isAllowedMagicBytes()` called before `upload()` on form submit; checks actual file bytes not Content-Type header |
| T-02-04-B | Elevation of Privilege | editToken in localStorage accessible to XSS | accept | App has no third-party scripts; invite-gated trusted audience; localStorage is the project's documented ownership mechanism (KEY DECISION in CLAUDE.md); no elevated server-side privilege is granted by possessing the token without Phase 3 verify endpoint |
| T-02-04-C | Tampering | Double-submit creating duplicate listings | mitigate | `isPending` guard at top of handleSubmit; submit button `disabled={isPending}` prevents UI re-trigger |
| T-02-04-D | Information Disclosure | editToken shown in UI | mitigate | editToken is never rendered in JSX — only written to localStorage; success banner shows "Your listing is posted!" with no token (D-02 / D-04) |
| T-02-04-E | Tampering | photoUrl forged by client (wrong blob URL) | accept | Invite-gate is the trust boundary — an authenticated friend submitting a URL from a different Blob store gets a broken image card, not a security breach; server does not verify Blob ownership |
| T-02-04-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed in this plan |
</threat_model>
