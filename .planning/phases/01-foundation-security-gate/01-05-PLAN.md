<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Local repo -> Vercel build environment | Trusted (Vercel pulls from git, runs build with project env vars) |
| Vercel build env vars -> drizzle-kit migrate -> Turso | Trusted; TURSO_AUTH_TOKEN must have DDL privileges |
| Internet -> Vercel Edge -> proxy.ts | Same boundary as Plan 01's threat model T-01-01 onwards |
| Vercel logs -> developer dashboard | Logs may contain INVITE_TOKEN in URL paths (RESEARCH.md Pitfall 7); accepted for Phase 1, hardened in later phase |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | Build-time migration failure leaves DB inconsistent | mitigate | drizzle-kit migrate exits non-zero on failure; Vercel aborts the build (deployment is not promoted); rollback is automatic |
| T-05-02 | Information Disclosure | Env vars leak to client bundle | mitigate | All seven env vars are server-only (no `NEXT_PUBLIC_` prefix); typed env loader (lib/env.ts) imports `server-only` indirectly via lib/db.ts and lib/session.ts |
| T-05-03 | Information Disclosure | INVITE_TOKEN visible in Vercel access logs | accept | Inherent to URL-path token design; rotation UI is Phase 3 (ADMN-04); flagged in RESEARCH.md Pitfall 7 |
| T-05-04 | Spoofing | secure: false cookie flag in production | mitigate | createAdminSession from Plan 03 conditions secure on `process.env.NODE_ENV === 'production'`; Vercel automatically sets NODE_ENV=production for the Production environment; verified in Task 04 acceptance |
| T-05-05 | Denial of Service | Cold-start timeout (default 10s on Hobby) | accept | Plan 01's shell page is a Server Component with no DB I/O; cold-start cost is Node init + proxy.ts execution + render — well under 10s |
| T-05-06 | Tampering | Vercel deployment of a branch / wrong commit | mitigate | Production deploys only from the main branch (Vercel default); manual `vercel --prod` requires the user's confirmation |
| T-05-07 | Repudiation | No deployment audit trail in app | accept | Vercel dashboard already provides deployment history; the app itself does not need an audit log for Phase 1 |
| T-05-SC | Tampering | Vercel CLI install (if used) | accept | Vercel CLI is installed locally per user_setup; npm registry verified package; out-of-scope for app dependency audit |
</threat_model>

    #!/usr/bin/env bash
    set -euo pipefail
    URL="${1:?usage: cold-start-verify.sh <production-url>}"
    TOKEN="${INVITE_TOKEN:?INVITE_TOKEN env var required}"
    FAIL=0
    check() {
      local label="$1" expected="$2" path="$3"
      local actual
      actual=$(curl -s -o /dev/null -w "%{http_code}" "$URL$path")
      if [ "$actual" = "$expected" ]; then
        echo "ok    $label  $path -> $actual"
      else
        echo "FAIL  $label  $path -> $actual (expected $expected)"
        FAIL=1
      fi
    }
    check "root-no-token-404"      404 "/"
    check "wrong-token-404"        404 "/wrong-token/"
    check "right-token-200"        200 "/$TOKEN/"
    check "admin-login-reachable"  200 "/$TOKEN/admin/login"
    check "admin-no-cookie-307"    307 "/$TOKEN/admin"
    check "static-favicon-200"     200 "/favicon.ico"
    BODY=$(curl -s "$URL/$TOKEN/")
    if echo "$BODY" | grep -q FriendSwap; then
      echo "ok    body-contains-FriendSwap"
    else
      echo "FAIL  body did not contain 'FriendSwap'"
      FAIL=1
    fi
    if echo "$BODY" | grep -q 'Nothing here yet'; then
      echo "ok    body-contains-empty-state"
    else
      echo "FAIL  body did not contain 'Nothing here yet'"
      FAIL=1
    fi
    if [ $FAIL -eq 0 ]; then
      echo
      echo "cold-start-verify: ALL CHECKS PASSED for $URL"
      exit 0
    else
      echo
      echo "cold-start-verify: FAILURES — see above"
      exit 1
    fi
