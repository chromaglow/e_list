<objective>
Build the working `POST /{INVITE_TOKEN}/api/upload` Route Handler that issues short-lived Vercel Blob client-upload tokens (`@vercel/blob/client` handleUpload pattern) with the security controls CLAUDE.md mandates: content-type restricted to JPEG/PNG/WEBP, 8 MB max, one file per upload, random suffix on pathnames. The route file lives at `app/[token]/api/upload/route.ts` — UNDER the `[token]` dynamic segment so Plan 01's proxy.ts gates it automatically (D-06; no proxy.ts modification needed; no admin session required because Phase 2 guests upload listing photos).

Magic-byte validation is implemented as a CLIENT-SIDE helper (`isAllowedMagicBytes`) exported from `lib/upload-validators.ts`. The @vercel/blob/client pattern transfers file bytes browser-to-CDN directly — the server only generates a short-lived token and never sees file content. The CLAUDE.md requirement "validate by magic bytes (not Content-Type)" is satisfied by having the browser read the file's first 12 bytes via FileReader/arrayBuffer and call `isAllowedMagicBytes()` BEFORE invoking the Vercel Blob `upload()` SDK. Phase 1 exposes the helper and demonstrates it in the smoke test; Phase 2 wires it into the actual listing form.

Purpose: Close the highest-risk integration point (D-04 — "delivers a working upload endpoint, not just env var configuration") so Phase 2 can layer listing creation on top without rediscovering the 4.5 MB Vercel function body limit (RESEARCH.md Critical Finding 2). Also lock in the client-side magic-byte contract so Phase 2's UI doesn't have to invent it.

Output: A POST /{token}/api/upload route handler that token-exchanges through @vercel/blob/client.handleUpload, the ALLOWED_TYPES/MAX_SIZE_BYTES/isAllowedMagicBytes module Phase 2 will reuse for the client-side pre-check, and a smoke-test script proving the route responds correctly when called with a HandleUploadBody AND demonstrating the client-side magic-byte check against known good/bad byte sequences.

Implements: D-04 (working upload Route Handler), D-05 (magic byte / 8 MB / 1 file controls — magic byte enforced client-side per the architectural constraint above), D-06 (invite-token gate only via [token] segment, no admin requirement).

**Wave 2 parallel contract:** This plan does NOT modify `lib/env.ts` or `proxy.ts`. Plan 01 declared `env.BLOB_READ_WRITE_TOKEN` getter and the proxy matcher already covers all `/{token}/*` paths. This plan only replaces the `BLOB_READ_WRITE_TOKEN=REPLACE_ME_in_plan_04` placeholder in `.env.local`.

**D-05 compliance note:** CLAUDE.md says "validate by magic bytes (not Content-Type)". The @vercel/blob/client architecture means the server NEVER sees the file bytes — `upload()` in the browser opens a direct channel to the Vercel Blob CDN using the short-lived token. Server-side magic byte enforcement is architecturally impossible in this pattern. We satisfy CLAUDE.md's INTENT (content-based validation, not Content-Type header trust) by validating magic bytes in the browser BEFORE the upload starts. Vercel Blob's `allowedContentTypes` in `onBeforeGenerateToken` is a second layer (Vercel enforces it on the CDN side as documented in RESEARCH.md Pattern 5). The full enforcement chain is: browser magic-byte check (lib/upload-validators.ts.isAllowedMagicBytes) -> Vercel CDN allowedContentTypes -> stored blob.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> /{token}/api/upload (token exchange) | Already gated by proxy.ts invite check (D-06); body is HandleUploadBody (signed handshake metadata, not the file itself) |
| Browser -> Vercel Blob CDN (direct upload) | Authorized by the short-lived token; Vercel enforces allowedContentTypes + maximumSizeInBytes server-side |
| BLOB_READ_WRITE_TOKEN env var | Server-side only; never sent to the browser; used by @vercel/blob/client to sign upload tokens |
| File content -> Vercel Blob storage | Object stored on `.blob.vercel-storage.com` (separate origin — CLAUDE.md requirement); pathname returned, full URL derived later |

export const ALLOWED_TYPES = ['image/jpeg', 'image/png', 'image/webp'] as const

 * Client-side magic-byte validator. CALL BEFORE @vercel/blob/client upload().
 * Browser pattern:
 *   const head = new Uint8Array(await file.slice(0, 12).arrayBuffer())
 *   if (!isAllowedMagicBytes(head)) { showError('Not a real JPEG/PNG/WebP image'); return }
 *   await upload(file.name, file, { access: 'public', handleUploadUrl: `/${token}/api/upload` })
 *
 * Recognizes:
 *   JPEG: FF D8 FF
 *   PNG:  89 50 4E 47 0D 0A 1A 0A
 *   WebP: 'RIFF' (52 49 46 46) at bytes 0-3 AND 'WEBP' (57 45 42 50) at bytes 8-11

 * The @vercel/blob/client architecture transfers file bytes BROWSER->CDN directly;
 * the server never sees file content, so server-side magic-byte validation is
 * architecturally impossible. CLAUDE.md's "validate by magic bytes (not Content-Type)"
 * intent is satisfied by browser-side enforcement here, plus Vercel CDN's
 * allowedContentTypes server-side enforcement as a second layer.
 */
export function isAllowedMagicBytes(firstBytes: Uint8Array): boolean
